# Design: Latest Flagship Roster Promotion

## Decision

“Latest” means the newest top-capability callable flagship in each existing council family, not the newest release-date SKU. GPT-5.6 Sol and Grok 4.5 replace their incumbents. Gemini 3.1 Pro Preview remains the advanced Gemini reasoning seat; Gemini 3.5 Flash remains a cost/latency challenger rather than a Pro replacement.

Compact mode is seats 1–5. Fable remains chairman. Evaluator priority is Fable, DeepSeek V4 Pro, GPT-5.6 Sol.

## Identity and compatibility

Promoted models receive production lifecycle, seats, roles, aliases, and reasoning settings. GPT-5.5 and Grok 4.3 lose production roles/seats and become legacy compatibility records. Explicit old logical IDs retain their existing routes. Aliases move to the promoted IDs. Cache keys and execution-plan digests continue to include the exact logical ID and captured route settings.

## Route policy separation

`allow_declared_route_failover` and `allow_provider_substitution` are independent execution settings. Declared failover means advancing to another explicit registry route after a recorded failure. Provider substitution controls OpenRouter's internal upstream substitution. Production MAY allow declared failover. Benchmarks and support probes MUST disable both. Every result records the attempted route, selected route, fallback flag, and normalized failure reason.

Grok 4.5 routes are:

1. `xai:x-ai/grok-4.5` using provider model ID `grok-4.5`.
2. `openrouter:x-ai/grok-4.5` using provider model ID `x-ai/grok-4.5`.

Fable production strict policy remains a single Vertex route and cannot be weakened by request, provider, constructor, preferred-route, missing-record, or ambiguous-route state.

## Benchmark and promotion evidence

No-fallback probes verify exact IDs before paid benchmark execution. The versioned public/non-PHI suite compares GPT-5.6 Sol against GPT-5.5 and Grok 4.5 against Grok 4.3, includes GPT evaluator behavior, and runs full-council seat ablations. Subjective scoring uses at least two independent judge families and objective validators where available.

Hard acceptance gates are exact route callability, zero undeclared provider substitution, deterministic projection drift checks, and successful rollback. The immutable representative 52-case live benchmark hard-gates only its evidenced quality, factual-error, objective, evaluator-format, route, latency, and cost thresholds: quality MUST be within three percentage points overall, no task stratum worse by more than five points, objective accuracy within two points, evaluator formatting no worse than baseline, full-council factual error no worse than baseline, at least 99% route success, less than 2% direct Grok failover, p95 full-council latency increase at most 20%, individual p95 at most 1.5× baseline, and cost increase at most 25% unless explicitly accepted. Its hashes, run identity, no-fallback policy, completeness, and passing threshold outcomes MUST be verified before any rollout read, snapshot, deploy, or traffic mutation.

The immutable benchmark owns the relative promotion decision. It binds the roster, registry digest, run manifest, policies, prompt suite, and threshold outcomes, but does not and cannot claim identity with an image SHA built later. Deployment probes instead bind runtime wiring: exact revision/image, registry and projection digests, roster, routes, payload schema, policy, and strict objective output. They do not rerun the benchmark or claim comprehensive safety evaluation.

The prospective top-level `run_rollout(...)` controller, not an entry-point caller, owns the fixed sequence: preflight benchmark → lock → prior snapshot validation → build → prior stream → deploy shadow/health → shadow sync/stream → 10% → planned restore → restarted 10% → 50% → authoritative 100% convergence → strict approved production health verification → final sync/stream → terminal cleanup/verification → durable retention obligation → mutation disarm → lock release. Its paid plan is not an argument and cannot be replaced or reordered. Fresh mode starts at zero and contains exactly five expected attempts: prior production stream, candidate shadow sync, candidate shadow stream, final production sync, and final production stream. Generic count carry-forward is rejected; only the exact evidence-bound continuation below may initialize the ledger at two. Any fresh retry consumes the remaining sixth slot and no mode can authorize a seventh request.

`resume-after-prior-v1` is the only continuation mode. It is an exact authorization tuple (mode, canonical count 2, approved-prefix manifest URI, exact generation), not a boolean. Python callers may supply only an object whose type is exactly `int`; the CLI may supply only canonical strings `"0"` or `"2"`. Booleans, floats (including integral-looking `0.0`/`2.0`), fractional values, and noncanonical strings fail closed. Every integer in a manifest, checkpoint, recovery proof, or source attestation must have JSON/Python type exactly integer, never merely compare equal to one. Before build it verifies benchmark, lock absence, immutable manifest/source generations, current URL and complete canonical traffic hashes, then CAS-acquires a lock binding manifest generation and target SHA, repeats all reads, proves unchanged one-owner production, and runs fresh prior health. Attempts 1-2 must come from two distinct source rollout IDs. Each source uses the exact fixed-bucket `rollout-evidence/<source-rollout-id>/attempts/000N-completed.json`, `rollout-evidence/<source-rollout-id>/recovery.json`, and `rollout-evidence/<source-rollout-id>/source-attestation.json` paths at exact generations. All four underlying checkpoint/recovery references and both attestation references are pairwise distinct. Each immutable source attestation has a strict no-extra-fields schema binding schema version 1, mode, source rollout ID and approved SHA, current target SHA, fixed namespace, exact checkpoint/recovery references, expected attempt number, service URL hash, canonical traffic hash, and prior revision. The manifest repeats each source ID and source approved SHA and references the exact attestation generation. The controller only reads and validates these post-merge attestations; it never creates or updates them. The resumed sequence starts at shadow sync as attempt 3; all infrastructure and semantic failures are terminal without retry. Fresh mode permits no manifest and preserves the normal five-probe sequence.

Production namespace is compiled as `PROJECT=tke-phi-privacy-engine`, `REGION=us-central1`, and `SERVICE=llm-council`. Conflicting environment, startup, controller-boundary, or adapter-constructor values fail before benchmark, lock, service access, build, or paid work. Revision identity is compared after strict normalization using only those constants: short Cloud Run revision IDs and only exact `projects/tke-phi-privacy-engine/locations/us-central1/services/llm-council/revisions/{id}` names are accepted. Boundary attributes cannot redefine normalization. This applies to latest-created, latest-ready, template, expected deploy, and revision-resource ownership paths.

Before each request the controller creates immutable `rollout-evidence/<rollout-id>/attempts/NNNN-started.json` with generation-match 0. After a classified result it separately creates immutable `NNNN-completed.json`; neither record is updated. The completion repeats and exactly matches rollout ID, attempt number, stage, surface, SHA-256 URL identity, optional retry-of presence/value, and paid-gate state from its start, then adds exactly one bounded outcome classification. Recovery accepts only a contiguous sequence of complete started/completed pairs whose JSON, exact schema/fields, types, values, filename numbers, record numbers, rollout IDs, and every repeated pair field all match. Gaps, non-contiguous numbering, completion without start, duplicates, malformed JSON/schema/fields, any field mismatch, and orphan starts all permanently close the paid gate, restore exact prior state if mutation is armed, stop the rollout, and make no further paid request. Records remain content-free and never contain content or raw errors. A single global retry MAY repeat only the same stage, URL, and surface after classified HTTP 429, HTTP 503, timeout, or connection reset. Semantic failures are terminal and non-retryable. No paid calls occur at either 10% stage, 50%, or during restoration. Planned restoration temporarily suppresses calls; terminal rollback permanently closes them.

## Deployment

### Temporary legacy prior-health bridge

`GcloudBoundaries.health()` is strict by default. A narrowly scoped `allow_legacy_prior=False` switch is set to true only for initial prior identity capture, partial-stage sampling, planned restoration proof, and terminal rollback proof. The switch delegates to the existing `health_identity` compatibility behavior, so only a healthy payload with a valid config and an entirely absent `artifacts` field is accepted; present empty, malformed, or partial artifacts still fail. Candidate shadow health and both final candidate verifications always use the strict deploy-health verifier. Prior and candidate identities must differ before any stage sample, and 100% samples remain candidate-only.

This compatibility bridge exists only because the currently retained pre-promotion revision predates artifact identity emission. Remove the switch and all true call sites after production and every retained rollback target emit the complete artifact contract.

Build from the reviewed merged Forgejo tree and pin the image digest plus registry/projection digests. GitHub is a mirror and its deploy workflow is manual `workflow_dispatch` only; dispatch requires an approved full Forgejo SHA and fails before build unless the input SHA, checkout SHA, and freshly fetched Forgejo `master` SHA are identical. Local deployment likewise fails before build or service access unless the worktree is clean and `HEAD` equals freshly fetched Forgejo `origin/master`. Both executable entry paths perform those checks before calling `run_rollout`. A mirror push to `master` cannot own a deployment. The operator executes at most one rollout, and any failed rollout stops without an automatic second attempt.

The rollout verifies the immutable benchmark, then atomically acquires the cooperative GCS lock at `gs://tke-phi-privacy-engine_cloudbuild/rollout-locks/tke-phi-privacy-engine/us-central1/llm-council.lock` with `--if-generation-match=0` before build or any service access. Existing or stale locks return explicit `stale_lock_recovery_required` and are never stolen, auto-expired, released, or treated as automatically recoverable by a normal run. Manual recovery is separate and requires durable snapshot and owned-state proof. It records the lock generation and deletes only that generation on terminal release. It rejects malformed UID, etag, generation, observed generation, ownership, traffic data, or service URI; requires the initial service to be converged; captures the exact HTTPS service base URL from that same initial Service response along with canonical traffic/tags, prior image, and health identity; and requires exactly one resolved prior revision at 100% before deploy. The captured URL is immutable rollout state and is the sole URL used for planned and terminal prior-health proof, including both already-restored/no-PATCH and PATCH rollback paths; no placeholder or later rediscovery is allowed.

Every traffic change uses Cloud Run v2 Service GET followed by a PATCH of only `traffic` carrying the captured service etag. After zero-traffic deploy, the controller also fetches the exact named candidate Revision resource and requires exactly one `Ready=CONDITION_SUCCEEDED`, exactly one `ContainerReady=CONDITION_SUCCEEDED`, and exactly one `Active` with state `CONDITION_FAILED`, severity `INFO`, and revision reason `RETIRED`. Only `ResourcesAvailable` and `MinInstancesProvisioned` may be omitted and each must match its exact retired shape when present; duplicate conditions, contradictory failed terminal or relevant conditions, and mismatched immutable image or ownership environment markers fail closed. Across initial validation, transition convergence, rollback convergence/proof, deploy ownership, and current-state ownership, omitted protobuf-JSON `reconciling` normalizes to false; explicit boolean false/true retain their meanings, while explicit null or any nonboolean value fails closed. Omitted traffic-target `percent` normalizes to zero only in Cloud Run traffic-target normalization; explicit null, noninteger, or out-of-range values fail closed. The rollout rejects malformed or error long-running operations, then polls realistic v2 Service state until `observedGeneration == generation`, reconciliation is false, and both canonical `traffic` and URI-normalized `trafficStatuses` exactly match the requested traffic and tags. UID MUST remain stable, generations MUST progress as expected, and each accepted transition MUST expose a fresh etag. A converged no-traffic service whose unique terminal Ready and ConfigurationsReady conditions succeeded with revision reason `RETIRED` is owned when latest-created/template/revision-resource/image/env/traffic evidence is exact even though `latestReadyRevision` remains the prior tagged revision. Each UID, generation, observed generation, reconciliation, etag freshness, traffic, tag, status, condition, and revision-resource dimension independently fails closed on mismatch rather than overwriting another operator.

Canonical traffic semantics are order-insensitive but target-exact: strict routing fields are normalized and sorted before equality or hashing, with no target merging or tag omission. Target identity is the exact `(revision, tag)` pair (including absent-tag semantics); target identities and nonempty tags are unique, and every complete allocation sums to exactly 100%. Each generated stage transforms the snapshot's sole positive prior target in place to the remainder while preserving its revision and tag identity; it never preserves that positive target and adds a second prior allocation. A no-traffic deploy owns the exact snapshot without a candidate traffic target. Every tagged stage explicitly requests Cloud Run's automatic untagged candidate-0 target and a distinct tagged candidate target at the stage percentage. A generated shadow-tag collision with any snapshot tag fails before build, deploy, traffic mutation, or paid request. Planned and terminal restore return to the exact snapshot, and final untagged 100% has exactly one candidate-100 target. Source attestations use immutable `source-attestation-<target-sha>.json` paths; this target-specific rule supersedes the earlier generic attestation-path shorthand.

Live Cloud Run `trafficStatuses` is compared through one strict expected-status projection wherever API statuses are checked against configured traffic: canonicalize and validate the complete traffic first, then remove only an untagged zero-percent target whose same revision also appears as a tagged target. Unrelated untagged zero-percent targets, tagged zero-percent targets, and every positive target remain exact; entries are never merged. Unknown fields, malformed values, duplicates, invalid allocation, missing retained targets, extra statuses, and wrong revision/tag/percent fail closed. This projection governs initial validation, transition convergence, no-traffic deploy ownership, current ownership where status is compared with traffic, final verification, and planned or terminal rollback convergence/proof. Direct comparisons between two independently observed status lists remain exact canonical comparisons. This live-status rule supersedes the earlier same-document shorthand that statuses exactly match unprojected requested traffic.

`resume-after-shadow-v1` is a separate, incident-only continuation and does not widen `resume-after-prior-v1`. It accepts only canonical count 4, the fixed `rollout-evidence/resume-after-shadow-v1/` manifest namespace, and an exact generation. The current merged Forgejo master SHA remains the controller authorization while the immutable manifest separately binds target SHA `000754f3c963c002b25a35f0b13a7c01a69510f9`; those SHAs may differ. The no-extra-fields manifest binds source run `000754f3c963-1783979097`, the exact generation-pinned prior manifest and fully revalidates its complete source chain using the target SHA, exact started/completed attempts 3 and 4, exact recovery generation `1783980208306954` with cumulative count 4, exact restored service UID/generation/etag/URL hash/traffic hash, and candidate revision `llm-council-000754f3c963-9b63e747`, digest `sha256:b34c651fcbc29f8b6491bfd8ecdf0d7abe7b954d814a750c7ea3a7621dec10c2`, and target SHA. All numeric JSON fields require exact integer type and all GCS references are generation-pinned. Evidence and current state are read before and after CAS lock acquisition and again around fresh prior health.

After the locked double-read, the controller creates a permanent generation-0 claim at the fixed shadow-resume claims path keyed by SHA-256 of manifest URI plus generation. The claim binds exact manifest URI/generation, controller SHA, target SHA, continuation ID, and canonical count 4. Existing claim means permanent replay refusal. The claim is never deleted, including pre-mutation failure and successful or failed rollback. Historical lock generation in recovery is evidence only and grants no authority. This mode performs no build or deploy. It validates the retained candidate Revision resource and immutable target identity, transitions directly from restored prior-100 to tagged candidate-100, requires candidate-only production samples and strict target-SHA/image health before paid access, consumes only attempt 5 final sync then attempt 6 final stream without retry, and then converges to untagged candidate-100 while preserving unrelated snapshot zero targets and tags. Pre-mutation failures release only the newly acquired lock; post-mutation rollback requires exact owned-state status projection and restores the exact snapshot.

After strict zero-traffic shadow health, the candidate receives the shadow sync and stream probes. It then progresses to 10%, performs an exact planned restoration rehearsal without paid calls, leaves the paid gate available, restarts at 10%, observes for five minutes, advances to 50%, observes for five minutes, and advances to authoritative 100% before final sync and stream probes through the production URL. Partial-stage service health samples are diagnostic: every response must be one of the two known identities, but observing only one known identity is allowed. At 100%, every service health sample must be the candidate. Control-plane state alone proves traffic percentages. Revision-attributed request count, 5xx, and latency telemetry and query failures are diagnostic only; absent organic traffic is `no_evidence`, never fabricated success or failure or an implicit gate.

One injected monotonic clock establishes a 30-minute promotion deadline. Build, deploy, operation polling, service convergence, observations, health, and paid requests each receive only the budget remaining when they start; every repeated operation and Service poll recalculates a strictly reduced remaining value. If time expires between two actual controller polls, no next promotion poll or operation may start. Rollback uses a new independent 300-second grace even when promotion time is exhausted and applies the same decreasing-budget rule to its polls and health proof. Handlers for `ERR`, `TERM`, `INT`, `HUP`, and guarded `EXIT` keep rollback armed after owned mutation through final verification and cleanup; before owned mutation they generation-conditionally release the owned lock without reading or patching traffic. Every outer workflow timeout exceeds 2100 seconds. Rollback uses no paid request and is not proven until exact canonical traffic/tags and URI-normalized statuses plus prior health identity are restored, UID is unchanged, generation is converged, and the lock remains held through proof. If current canonical traffic and statuses already exactly equal the snapshot, rollback proves prior health and generation-conditioned lock ownership, emits no traffic PATCH, disarms mutation, and releases only the owned lock generation. Otherwise it retains the exact PATCH/LRO/fresh-generation/fresh-etag rollback contract. Each dimension has an independent fail-closed negative contract.

Success requires candidate-only untagged 100% traffic, exact candidate identity and digests, converged generation, removal of only the rollout-created shadow tag, and preservation of unrelated tags. Strict production health binds the approved SHA, immutable image, registry/projection digests, roster, routes, and Vertex policy after authoritative 100% convergence and before final sync/stream. Terminal order is final verification, durable content-free GCS retention obligation creation, rollback-mutation disarm, generation-conditioned lock release, then guarded exit with no mutation after release. The obligation contains rollout ID, prior revision, and `retain_until`, derived no earlier than the injected wall clock's current UTC time plus 24 hours. Deployment runtime never executes `bd`; after success, the orchestrator creates the time-blocked follow-up Bead from that obligation. The rollout never deletes a revision. `scripts/deploy.sh` is the sole production entry point and delegates the whole operation to the controller. GitHub CI and deploy jobs both checkout and verify the identical full approved Forgejo SHA. GitHub manual dispatch calls only that entry point after SHA checks. `cloudbuild.yaml` is build-only or delegates to the same controller and cannot independently mutate Cloud Run or issue paid probes. This contract is prospective and does not claim implementation or deployment has completed.
