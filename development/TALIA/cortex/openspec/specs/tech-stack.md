#### Why MySQL Over PostgreSQL

PostgreSQL has known connection pool stability issues with AppSheet. AppSheet opens a connection per table during sync, and connections persist for minutes. PostgreSQL's process-per-connection architecture leads to "too many connections" errors with as few as 3-4 concurrent AppSheet users. PgBouncer mitigates but introduces its own issues ("Exception while reading from stream"). MySQL's thread-per-connection model handles AppSheet's chatty sync pattern natively.

CORTEX does NOT require PostgreSQL-specific features:
- **AI processing** → Vertex AI + Cloud Run (Python), not in SQL
- **Vector search** → Vertex AI RAG Engine (Spanner-backed), not pgvector
- **JSONB** → MySQL 8.4 JSON type is sufficient (store/retrieve blobs, no in-DB JSON path queries at scale)
- **CTEs/Window functions** → MySQL 8.0+ supports these
- **Performance** → Negligible difference at TKE scale (~15-20 concurrent users)

## 10. Authentication & Authorization

### Identity-Aware Proxy (IAP) — Zero-Trust Authentication

CORTEX uses **Google Cloud Identity-Aware Proxy (IAP)** as the primary authentication layer. IAP enforces authentication at the infrastructure level — every request is verified before it reaches the application. No application-level OAuth flow is needed.

```
┌─────────────────────────────────────────────────────────────────┐
│                     REQUEST FLOW                                 │
│                                                                  │
│  Browser/PWA                                                     │
│       │                                                          │
│       ▼                                                          │
│  ┌──────────────────────────────────────────────────┐           │
│  │           Identity-Aware Proxy (IAP)              │           │
│  │                                                    │           │
│  │  1. Checks Google session cookie                   │           │
│  │  2. If not signed in → redirect to Google Sign-In  │           │
│  │  3. Verify @thekidneyexperts.com identity          │           │
│  │  4. Enforce Context-Aware Access policies          │           │
│  │  5. Inject signed JWT header                       │           │
│  │     (X-Goog-IAP-JWT-Assertion)                     │           │
│  └──────────────┬───────────────────────────────────┘           │
│                  │                                                │
│                  ▼                                                │
│  ┌──────────────────────────────────────────────────┐           │
│  │              Cloud Run (FastAPI)                   │           │
│  │                                                    │           │
│  │  1. Verify IAP JWT signature (ES256)               │           │
│  │  2. Extract user email + sub from JWT              │           │
│  │  3. Look up RBAC role in Cloud SQL                 │           │
│  │  4. Enforce authorization (who can do what)        │           │
│  │  5. Serve request                                  │           │
│  └──────────────────────────────────────────────────┘           │
│                                                                  │
│  IAP = Authentication (who are you?)                             │
│  App = Authorization (what can you do?)                          │
└─────────────────────────────────────────────────────────────────┘
```

| Property | Value |
|----------|-------|
| **Authentication Layer** | Google Cloud Identity-Aware Proxy (IAP) |

| **Identity Provider** | Google Workspace (@thekidneyexperts.com) |

### IAP JWT Verification (FastAPI Middleware)

```python
from google.auth.transport import requests as google_requests
from google.oauth2 import id_token
from fastapi import Request, HTTPException

IAP_AUDIENCE = "/projects/PROJECT_NUMBER/global/backendServices/BACKEND_ID"

async def verify_iap_jwt(request: Request) -> dict:
    """Verify IAP JWT and extract user identity."""
    iap_jwt = request.headers.get("X-Goog-IAP-JWT-Assertion")
    if not iap_jwt:
        raise HTTPException(status_code=401, detail="Missing IAP JWT")
    
    try:
        claims = id_token.verify_token(
            iap_jwt,
            google_requests.Request(),
            audience=IAP_AUDIENCE
        )
        return {
            "email": claims["email"],       # user@thekidneyexperts.com
            "google_id": claims["sub"],      # Stable unique identifier
        }
    except ValueError as e:
        raise HTTPException(status_code=401, detail=f"Invalid IAP JWT: {e}")
```

### Context-Aware Access Policies

IAP enforces these policies at the infrastructure level before any request reaches the app:

| Policy | Target | Rule | Rationale |
|--------|--------|------|-----------|
| **IP allow-list** | `data_entry` | Allow only from Citrix server IP ranges | Philippines team must use Citrix VDI |
| **US geography** | All roles | Restrict to US-origin requests | All PHI access from US infrastructure |
| **Working hours** | `data_entry` | Restrict to defined hours (e.g., 6 AM - 6 PM CST) | Limit data entry access window |
| **No device restriction** | `provider` | Allow personal phones + managed Chromebooks | Providers need 24/7 mobile access (on-call) |
| **No time restriction** | `provider`, `scribe` | 24/7 access | Clinical needs don't follow a schedule |

**Note on Citrix VDI:** IAP sees the Citrix server's US IP address, not the Philippines client IP. This is correct — PHI access originates from US infrastructure.

**Note on Chrome Enterprise:** TKE has Chrome Enterprise Standard. IAP device trust enforcement (requiring managed devices only) would require Chrome Enterprise Premium (~$6/user/month) and is NOT used — providers need personal phone access.

### Signed GCS Upload URLs (Offline Audio Sync)

Service workers may not have access to IAP session cookies. To support offline audio recording + background upload, CORTEX uses **pre-signed GCS upload URLs**:

```
┌─────────────────────────────────────────────────────────────────┐
│                OFFLINE AUDIO UPLOAD FLOW                          │
│                                                                  │
│  WHILE ONLINE (IAP session active):                              │
│  1. App requests signed upload URLs from backend                 │
│  2. Backend generates PUT-only signed GCS URLs                   │
│     - Scoped to: recordings/{encounter_id}/{recording_id}.flac  │
│     - Expiry: 4 hours                                            │
│     - Method: PUT only (no GET, DELETE, LIST)                    │
│  3. URLs cached in service worker / IndexedDB                    │
│                                                                  │
│  WHILE OFFLINE:                                                  │
│  4. Audio recorded to IndexedDB (encrypted with session key)    │
│                                                                  │
│  WHEN BACK ONLINE:                                               │
│  5. Service worker uploads audio directly to GCS via signed URL │
│     - Bypasses IAP (direct GCS upload)                           │
│     - Encrypted in transit (TLS 1.3)                             │
│     - Encrypted at rest by GCS (AES-256)                         │
│  6. Backend notified via GCS Pub/Sub → starts transcription     │
│  7. Local encrypted copy deleted from IndexedDB                  │
└─────────────────────────────────────────────────────────────────┘
```

```python
from google.cloud import storage
from datetime import timedelta

def generate_signed_upload_url(encounter_id: str, recording_id: str) -> str:
    """Generate a PUT-only signed URL for audio upload to GCS."""
    client = storage.Client()
    bucket = client.bucket(f"cortex-audio-{ENV}")
    blob = bucket.blob(f"recordings/{encounter_id}/{recording_id}.flac")
    
    url = blob.generate_signed_url(
        version="v4",
        expiration=timedelta(hours=4),
        method="PUT",
        content_type="audio/flac",
    )
    return url
```

**Security properties of signed upload URLs:**
- Time-limited (4-hour expiry)
- Scoped to a single GCS object (cannot access anything else)
- PUT-only (cannot read, list, or delete)
- Unforgeable (HMAC-SHA256 or RSA signed)
- Generated only by IAP-authenticated backend (verified user)
- PHI (audio) encrypted in transit (TLS) and at rest (GCS AES-256)
- GCS covered under TKE's GCP BAA — fully HIPAA compliant

### WebSocket & SSE Considerations with IAP

| Protocol | IAP Behavior | Mitigation |
|----------|-------------|------------|
| **WebSocket** (live transcription) | Authenticates initial HTTP Upgrade only. No ongoing auth checks. | Acceptable — encounters are 5-15 min, IAP sessions last hours. Client reconnection logic handles drops. |
| **SSE** (council generation progress) | Works but may buffer responses. | Use HTTP 200 status, set `X-Accel-Buffering: no` and `Cache-Control: no-cache` headers. Polling fallback if needed. |

### Role-Based Access Control (RBAC)

IAP handles **authentication** (who are you?). The application handles **authorization** (what can you do?) via RBAC stored in Cloud SQL:

| Role | Permissions |
|------|-------------|
| `provider` | Full read/write, sign notes, approve billing |
| `scribe` | Create/edit notes, manage recordings, submit for review |
| `data_entry` | Paste EPIC data, upload screenshots, confirm parsed data |
| `admin` | All above + user management, audit log access, system config |
| `qa` | Read-only access to signed notes, quality metrics |

### Authorization Enforcement

- **IAP** for infrastructure-level authentication (zero-trust perimeter)
- **Context-Aware Access** for IP, geographic, and time-based policies
- **Application-level RBAC** stored in Cloud SQL (role → permissions)
- **Row-level security** for patient data (provider can only see assigned patients)
- **Cloud IAM** for GCP resource access (service accounts, buckets, etc.)

### PHI Boundary Model

| Scenario | Audio Behavior |
|----------|---------------|
| **Encounter active** | Omi webhook streams to CORTEX. Audio tagged to encounter. Processed by CORTEX STT. |
| **Between encounters** | Audio stays in Omi's HIPAA-compliant cloud only. Does NOT enter CORTEX. |
| **Retroactive pull** | Provider explicitly selects an Omi conversation segment and associates it with an encounter. Manual trigger preserves intentionality. |
| **Multi-patient hallway** | Provider chooses which Omi segment maps to which patient. CORTEX never auto-ingests ambient audio. |

### Real-time Audio Webhook Endpoint

Omi streams raw PCM16 audio bytes to CORTEX during active encounters:

| Property | Value |
|----------|-------|
| **Endpoint** | `POST /api/audio/omi` |
| **Content-Type** | `application/octet-stream` |
| **Audio Format** | Raw PCM16 (16-bit signed, little-endian) |
| **Sample Rate** | 16,000 Hz |
| **Channels** | Mono (1) |
| **Chunk Interval** | Every 10 seconds (configurable in Omi app) |
| **Query Params** | `sample_rate`, `uid` (Omi user identifier) |
| **Auth** | HMAC signature verification + mapped to CORTEX user |

```python
# FastAPI endpoint for Omi audio webhook
@app.post("/api/audio/omi")
async def receive_omi_audio(
    request: Request,
    uid: str = Query(...),
    sample_rate: int = Query(default=16000),
):
    """Receive real-time audio chunks from Omi device."""
    audio_bytes = await request.body()
    
    # Map Omi uid to CORTEX user
    user = await get_user_by_omi_uid(uid)
    if not user:
        raise HTTPException(404, "Unknown Omi user")
    
    # Find active encounter for this user
    encounter = await get_active_encounter(user.id)
    if not encounter:
        # No active encounter — discard (ambient stays in Omi cloud)
        return {"status": "no_active_encounter", "discarded": True}
    
    # Buffer audio and route to STT pipeline
    await audio_buffer.append(
        encounter_id=encounter.id,
        audio_bytes=audio_bytes,
        sample_rate=sample_rate,
        source="omi_stream",
    )
    return {"status": "ok"}
```
